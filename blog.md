# 用 Codex 把 Markdown 简历做成 PDF：我的开源化实践

## 为什么要把这个项目开源

我之前一直把这套简历流程当作个人工具使用：先整理一份 Markdown 简历，再根据自己喜欢的视觉风格生成 HTML，最后导出 PDF。流程并不复杂，但很适合交给 Codex 这类代码代理来完成。

这次把它整理成开源项目，目标不是做一个功能庞杂的简历平台，而是保留一个足够轻量、足够直接的工作流：

- 用户提供简历内容，或者让 Codex 先生成 `profile.md`
- Codex 参考项目里的模板和设计说明生成 HTML
- 再用 Chromium headless 输出 PDF

对于想快速做出一份可投递简历的人来说，这种方式的好处是明确的：文本、样式、导出链路都在本地，改起来快，也不依赖复杂后端服务。

## 这个项目现在是什么结构

项目结构很简单：

- `resume_design_summary.md`：记录模板的设计思路和视觉特征
- `examples/profile.md`：示例 Markdown 简历
- `examples/resume.html`：示例模板
- `examples/resume-llm.html`：另一份示例模板
- `output/`：实际生成结果的统一输出目录
- `AGENTS.md`：约束 Codex 的生成流程

我这次开源化主要做了几件很克制的事情：

1. 把样例数据做了脱敏，去掉真实姓名、电话、邮箱、学校等信息。
2. 补上 `AGENTS.md`，明确要求 Codex 先阅读设计说明，再参考 `examples/` 生成最终简历。
3. 补上 `README.md`，把使用方式写清楚，让第一次打开项目的人知道怎么直接开始。
4. 使用 MIT 协议，降低复用门槛。

这几个动作本身都不大，但足以把“个人项目”变成“别人也能直接拿来用的项目”。

## 为什么我选择 Codex 来完成这件事

这个项目的核心不是传统意义上的“功能开发”，而是把已有素材组织成稳定工作流。Codex 在这件事上很合适，原因有三个：

- 它能直接理解仓库里的上下文，比如设计说明、示例文件和输出要求。
- 它适合做结构化生成，例如先产出 `profile.md`，再生成 HTML，最后补全导出文件。
- 它能直接在项目内执行命令，省去很多手动重复操作。

也就是说，这不是“AI 帮我写一段文案”这么简单，而是“AI 按项目约定完成一整条本地工作流”。

## 实际使用时怎么做

实际使用非常直接：

1. 在项目根目录打开 Codex。
2. 把自己的简历内容粘贴进去，或者让它先生成 Markdown 简历。
3. 告诉它参考 `resume_design_summary.md` 和 `examples/`。
4. 让它把结果输出到 `output/` 下，包括 `profile.md`、HTML 和 PDF。

如果已经有 HTML 文件，也可以直接使用 Chromium 的 headless 模式导出 PDF。这种方式的优点是稳定、可重复，而且很适合纳入自动化流程。

## 这类项目的价值不在“炫技”

我越来越觉得，很多 AI 项目真正有价值的地方，不在于堆多少复杂能力，而在于能不能把一个原本零散、手动、易出错的流程，收束成一个清晰的工作路径。

这个简历项目就是一个很典型的例子：

- 输入是 Markdown
- 中间产物是 HTML
- 最终结果是 PDF
- 执行者是 Codex
- 约束来自仓库内的文档和样例

这条链路足够小，但也足够完整。对于个人开发者来说，这种“小而完整”的项目，反而最适合真正落地。

## 结语

如果你也想把自己的简历制作流程整理成一个可复用、可迭代、可公开的项目，这种做法很值得试一次。先别急着做大，先把输入、模板、导出和约束写清楚，让 Codex 能稳定接手，你就已经拥有了一个实用的开源工作流。
